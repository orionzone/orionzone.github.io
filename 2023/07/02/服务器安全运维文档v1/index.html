服务器安全运维文档v1

梳理了一些服务器安全运维规范

一 服务器安全概述

服务器安全可分为三个方面，即保密性、完整性、自主性。

1.1 保密性

指保护服务器中存储的敏感数据不被未经授权的人员或系统所访问、查看和获取。保密性包括以下方面：

  • 人员授权：建立严格的人员访问策略，做好对用户身份的认证和鉴权，限制未授权人员访问数据。
  • 加密：对敏感数据进行加密，提高数据的安全性。

1.2 完整性

指确保服务器中的数据不被篡改、损坏或丢失。完整性包括以下方面：

  • 访问控制：限制数据访问权限，防止数据被篡改。
  • 备份和恢复：建立备份策略，定期备份数据并测试备份恢复流程。

1.3 自主性

服务器安全的自主性指服务器不被黑客用来挖矿、防止被黑客用来发起DDoS攻击、存储和分发恶意软件等。自主性包括以下方面：

  • 定期更新软件和系统：定期更新服务器操作系统和应用程序，修复已知的漏洞。
  • 安全加固：加强服务器网络和软件的安全性，尽可能降低攻击面。
  • 日志审计：定期检查服务器日志，及时发现异常行为并采取相应的措施。
  • 加强安全意识教育：加强员工的安全意识教育，提高其对网络安全的认识，减少员工操作不当造成的安全风险。

下面将从服务器用户、服务器网络、服务器软件、内部员工等多个潜在的引发服务器安全问题的入口出发来加强服务器的保密性、完整性和自主性。

二 服务器用户导致的安全问题

如果服务器用户在管理时存在着一些不当的操作，可能会引发一系列的安全问题，如运维人员在操作服务器时误输入了危险的指令或是对服务器权限等管理不善等，故下面将从运维误操作和身份被盗用两个方面来分析并解决此类安全问题。

2.1 运维误操作

运维过程中可能会无意或因不熟悉触发一些不希望发生的操作，应提前预防这些隐患。

2.1.1 误重启服务器

执行 Ctrl+Alt+Del 触发服务器重启。
解决方案：
禁用控制台 Ctrl+Alt+Del 组合键重启。

2.1.2 误删除文件

使用 rm 命令删除了不应删除的文件。
解决方案：

  • 禁用  rm -rf /
  • 建立回收站，rm操作实际将文件移动到  ~/.trash  ,并记录删除时间，误删可以找回。

2.2 身份被盗用

指黑客通过各种手段获取服务器用户的账户和密码，进而冒充该用户身份进行非法操作。黑客盗用服务器账户身份后，可以获取服务器中存储的敏感数据、控制服务器，甚至通过服务器发起攻击。以下是一些黑客盗用服务器账户身份的常见手段：

  • 暴力破解：黑客使用暴力破解工具，通过尝试不同的用户名和密码组合，猜测服务器用户的账户和密码。
  • 社交工程：黑客通过伪装成正常用户，使用各种方式获取用户账户和密码，如钓鱼邮件、假冒网站等。
  • 恶意软件：黑客通过植入恶意软件，监控用户的输入操作，获取账户和密码。

解决方案：
通过以下内容管理好用户权限，增加ssh登录的限制，并培训员工保护好自己的密钥文件。

2.2.1 用户权限管理

权限管理的作用是确保系统和数据安全，以及为用户提供访问特定资源的权限。生产环境中应当对系统进行安全性分级，使得管理和使用者之间的权限能够有效的划分，每个使用者的权限都应当被明确的定义，并且要求系统的变更必须经过审批。通过对Linux用户和用户组的合理管理，可以提高系统的安全性和稳定性，防止系统遭受攻击或误操作。

用户和用户组

用户：
Linux系统中的每个文件或者文件夹，都有一个所属用户及所属组，使用id命令可以显示当前用户的信息，使用passwd命令可以修改当前用户密码。Linux操作系统用户的特点如下：

  • 每个用户拥有一个UserID，操作系统实际读取的是UID，而非用户名；
  • 每个用户属于一个主组，属于一个或多个附属组，一个用户最多有31个附属组；
  • 每个组拥有一个GroupID；
  • 每个进程以一个用户身份运行，该用户可对进程拥有资源控制权限；
  • 每个可登陆用户拥有一个指定的Shell环境。

用户组：
所有的Linux或者Windows系统都有组的概念，通过组可以更加方便的管理用户，组的概念应用于各行行业，例如企业会使用部门、职能或地理区域的分类方式来管理成员，映射在Linux系统，同样可以创建用户，并用组的概念对其管理。
Linux组有如下特点：

  • 每个组有一个组ID；
  • 组信息保存在/etc/group中；
  • 每个用户至少拥有一个主组，同时还可以拥有31个附属组。

2.2.3 防利用服务器软件提权

利用服务器软件提权的常见手段：

  • 利用软件漏洞：攻击者通过利用服务器软件中的漏洞，获取管理员或root权限。
  • 缓冲区溢出：攻击者通过输入超出程序预设缓冲区范围的数据，使程序崩溃或产生意外行为，从而获取管理员或root权限。
  • 木马程序：攻击者通过在服务器中植入木马程序，获取管理员或root权限。

解决方案：

  • 加强文件上传的安全控制：建立上传文件的白名单，只允许上传特定类型的文件，防止黑客上传恶意文件。
  • 加强文件上传的监控：建立文件上传监控机制，定期检查上传文件，及时发现和处理异常行为。
  • 加强文件访问控制：限制文件访问权限，只允许授权用户访问文件，并采用访问审计系统监控访问行为。
  • 及时升级服务器软件，特别是出现漏洞时。
  • 在部署到生产环境前进行安全测试。
  • 更多服务器软件安全相关内容见  四 服务器软件导致的安全问题  。

三 服务器网络导致的安全问题

3.1 信任了非法的根证书

信任了非法的根证书是一种非常危险的情况，可能导致系统受到中间人攻击，从而导致敏感信息泄露、数据篡改等问题。为了防止信任了非法的根证书，可以采取以下措施：

  • 使用公开可信的根证书：使用公开可信的根证书，例如VeriSign、DigiCert等。这些根证书是由公开的第三方机构颁发的，可以保证其真实性和可靠性。
  • 禁止信任未知的根证书：禁止信任未知的根证书，只信任公开可信的根证书。如果遇到未知的根证书，应该先验证其真实性，再决定是否信任。
  • 定期更新证书：定期更新证书，及时更新受信任的根证书，防止其过期或被吊销。
  • 使用数字证书验证：使用数字证书验证，验证服务器是否具有正确的数字证书。数字证书可以保证服务器的真实性和可靠性，有效防止中间人攻击。
  • 使用加密连接：使用加密连接，例如HTTPS等，可以有效防止中间人攻击，保障通信安全。
  • 定期审计证书：定期审计证书，及时发现异常情况，并进行处置。例如，如果发现证书被吊销或存在伪造等情况，应及时进行处理。

在Ubuntu系统中，安装的根证书存储在以下目录中：
/etc/ssl/certs/
输入命令  sudo cat <证书名称>  ，查看指定证书的详细信息，例如  sudo cat ca-certificates.crt  。

3.2 使用未知的 dns 服务

使用未知的DNS服务可能会造成以下危害：

  • 未知的DNS服务可能会对用户的设备和数据造成安全威胁，例如中间人攻击、DNS欺骗等。
  • 未知的DNS服务可能会存在不稳定的情况，导致用户访问网络速度变慢，甚至无法正常访问网络。

3.3 服务默认端口直接对外

将服务默认端口直接对外开放可能会造成以下危险：

  • 默认端口可能会存在已知漏洞，黑客可以利用漏洞对服务器进行攻击和入侵，窃取敏感信息。
  • 服务端口直接对外开放可能会增加访问压力，导致服务器崩溃或者无法正常工作，影响业务的正常运行。

为了避免以上问题，应该采取以下措施：
更改默认端口：将服务端口修改为非默认端口，降低黑客攻击的成功率。
限制访问权限：仅允许必要的用户或者IP地址访问服务，降低攻击的风险（详见  2.2.2防非法ssh登录  ）。
使用防火墙：通过配置防火墙规则，限制端口对外开放，提高服务器的安全性。
在Ubuntu系统中，可以通过以下步骤进行端口限制：
输入命令  sudo ufw enable  ，启用防火墙。
输入命令  sudo ufw allow <端口号>/tcp  ，允许指定端口对外开放。
输入命令  sudo ufw reload  ，重新加载防火墙规则。

3.5 关键服务鉴权步骤薄弱

如果关键服务没有做好鉴权，可能会导致：

  • 未经授权的用户或进程可以访问敏感数据或执行危险操作，例如修改或删除重要文件、篡改数据库或执行恶意代码
  • 恶意软件或黑客可以利用软件漏洞进行攻击或利用，例如通过SQL注入、远程命令执行等攻击方式进入系统并窃取数据、加密数据勒索

因此有必要强制启用软件鉴权功能，并限制访问控制规则，确保只有授权的用户或进程才能执行特定的操作，并且应采用最小化原则，只安装必要的软件和服务，并删除不需要的软件和服务，以减少潜在的漏洞和攻击面。例如开发人员希望进入Kubernetes集群查看日志，则应配置好相应的RBAC鉴权策略而不是直接将管理员的.kube/config拷给开发人员。

四 服务器软件导致的安全问题

4.1 专门编写的恶意软件

Linux服务器一直给我们的印象是安全、稳定、可靠，性能卓越。由于一来Linux本身的安全机制，Linux上的病毒、木马较少，二则由于宣称Linux是最安全的操作系统，导致很多人以为它永远不会感染病毒、木马，以为它没有安全漏洞。然而近年来很多数据中心成为挖矿病毒重点攻击目标，所以不能忽视病毒对系统的影响。
如果用性能分析工具发现服务器异常怀疑有病毒存在，可以关注以下这些点：

  • 通过 crontab -l 查看是否有可疑的计划任务
  • 是否存在导致文件无法删除的chatter锁
  • 是否存在守护进程
  • 是否存在开机自启脚本
  • /root/.ssh/里面是否留有攻击者的秘钥

发现异常后kill掉相应进程并删除文件。
也可以尝试用ClamAV查杀（详见常用命令查询文档）。

4.2 经常爆漏洞的高危软件

禁止安装经常爆漏洞的高危软件。
如phpMyAdmin，wordpress等，如需安装软件需得到审批。

4.3 非法破解的软件

由于非法破解软件的来源和代码无法验证，存在恶意代码或被植入过第三方后门的风险，容易导致服务器被攻击，所以不应安装此类软件。

4.4 安装过多的软件

过多的软件可能导致服务器资源被耗尽，或者其中某个软件出现漏洞影响整个系统的安全性。所以安装软件前应关注服务器的资源状态，可以通过安装Prometheus和Node-Exporter监控服务器节点资源状态。

4.5 旧版本的软件

运维人员未及时更新软件，导致服务器存在安全漏洞或性能下降。但是升级应在得到了相应授权后进行，不得私自升级。
解决方案：
注意apt源安全，使用安全的源如  mirrors.tuna.tsinghua.edu.cn
Ubuntu换源修改  /etc/apt/sources.list
更新软件源  apt-get update
更新升级所有软件  apt-get upgrade
更新某个软件  apt-get upgrade 名
列出可更新的软件  apt list --upgradable

4.6 自制的软件

自制软件可能存在代码缺陷或漏洞，如果没有经过充分的测试和审核，可能会导致服务器安全问题。在部署应用时采用ci/cd流水线自动化部署更为安全，但是也需要注意相应的安全问题。（详见  4.7ci/cd流水线软件安全  ）

4.7 ci/cd流水线软件安全

4.7.1 Docker镜像

  • 使用可信赖的 Docker 基础镜像，通常情况下，它们在 DockerHub 官网具有明确的标识。
  • 尽量使用较小的基础镜像，最大程度减小安全攻击面。
  • 在编写 Dockerfile 的时候，使用非 root 用户来启动业务程序，可以借助开源工具 hadolint 来检查 Dockerfile 是否符合最佳实践，在VSCode上也有hadolint的扩展，本机安装完hadolint的可执行文件后，vscode里装上这个扩展就可以直接使用。
  • 使用  docker scan  命令来扫描构建的业务镜像，它内置非常多安全规则，可以最大程度避免出现安全问题。（https://docs.docker.com/engine/scan/  ）

4.7.2 业务依赖

  • 使用  docker sbom  命令来分析第三方依赖。
  • SBOM 清单只是用来记录软件包含哪些依赖组件的，你还需要通过 https://osv.dev/  开源的分布式漏洞数据库来查看 SBOM 内已知的组件漏洞，或者借助 Google 开源的 osv-scanner 项目来批量查询。

4.7.3 Docker镜像仓库

  • 不能把公共的镜像仓库当成集群的可信的镜像源。使用 Harbor 来搭建私有镜像仓库，并将它作为组织内部的唯一可信镜像源，同时定期进行漏洞扫描。以此来最大程度确保镜像仓库的可信和安全性。
  • 不使用latest，而使用类似 v1.0 或者 Git commit ID 作为镜像版本号。其次，镜像版本不应该能够被覆盖。

4.7.4 Git仓库

  • 将 Main 分支作为部署源。任何人将代码提交到 Main 分支时，都需要经过成员的人工 Review 才能合并，这样就可以有效防范恶意的镜像或者代码部署到生产环境中。
  • 关注在 Git 仓库中存储的明文密钥，例如镜像拉取凭据、数据库连接信息和外部服务凭据。应该通过加密的方式存储它们，如使用sealed-secrets。

4.8 云环境下的软件安全

采用k8s安装上述软件可以降低一定的安全风险，因为k8s具有以下安全特性：

  • 隔离性：k8s采用容器技术，可以实现应用之间的隔离，降低攻击面。
  • 滚动升级：k8s支持滚动升级，可以保证应用在升级过程中不会出现宕机等故障。
  • 自我修复：k8s具有自我修复能力，可以在出现故障时自动重启容器或重新部署应用。
  • 访问控制：k8s支持访问控制，可以限制用户或服务账号的访问权限。

但是，即使采用k8s安装这些软件，管理员也需要时刻关注这些软件的安全性和漏洞公告，并采取相应的措施来保护服务器和系统的安全。同时，应定期更新这些软件的最新版本，关闭或删除不必要的服务和端口，以减少服务器的攻击面。
如以下方面：

  • 仅限在内网访问 Kubernetes。
  • 通过 Kubernetes RBAC 提供集群访问凭据，而不是使用管理员的 Kubeconfig，根据“最小权限”的原则对资源和操作赋予权限。
  • 隔离 Kubernetes 节点而不是将它们暴露在公网当中。
  • 开启 Kubernetes 的审计日志，有助于及时发现与 API Server 的异常连接。

五 内部员工导致的安全问题

5.1 内部员工的不安全行为示例

  • 服务器密码贴在办公桌便利贴上。
  • 登上服务器后不锁屏电脑就离开办公桌。
  • 和同事关系好，把自己 root 权限的用户名和密码借他用用。
  • 公司楼下捡到一个 U 盘，插到办公室电脑上看看里面有什么。
  • 办公邮箱收到一个邮件，不确认发件人就填写资料领取公司福利。
  • 私自搭建代理，从家里或者公网穿透到办公内网，突破网络环境隔离。
  • 未经授权的人员可以通过物理入口进入服务器房间或数据中心，通过操纵服务器硬件或使用外部设备获取服务器数据或访问服务器系统。
  • 攻击者可以利用社交工程手段，通过欺骗内部员工，获得服务器访问权限或敏感信息。
  • 在服务器上运行不安全的程序或脚本，导致服务器受到攻击。
  • 使用弱密码或同一密码在多个系统中使用，导致账户被黑客攻破。
  • 对服务器进行未经授权的更改或升级，导致系统稳定性和安全性受到影响。
  • 使用未经授权的软件或未经验证的软件源，导致服务器受到攻击或感染恶意软件。
  • 在服务器上安装不必要的软件或服务，导致系统安全性降低。
  • 使用不安全的网络传输协议，如明文传输的FTP协议、Telnet协议等，容易被黑客窃取账户密码。
  • 在服务器上存储不安全的数据，如明文密码、客户信息等，容易被黑客攻击或泄露。

5.2 对内部员工行为的防范

首先应按上文做好相应的授权以及安全加固，内部员工应遵守企业的安全政策和规范，确保所有的操作都是合法和安全的。例如，不应该试图绕过安全措施，如使用非法的工具、共享密码、泄露机密信息等，同时要进行对内部员工的操作审计，及时发现问题（详见  5.3对内部员工的操作审计  ）。

5.3 对内部员工的操作审计

对内部员工的操作审计可以通过Linux自带的日志系统和auditd来实现。

系统日志

系统日志是记录系统硬件检查、内核动作、软件启动、用户动作等各项信息的文件。通过系统日志可以判断系统健康状态、检测系统问题、查找攻击证据等。运维人员未及时记录日志或删除日志，会导致无法快速发现和处理安全事件。
